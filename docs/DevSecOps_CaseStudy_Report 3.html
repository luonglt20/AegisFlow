4. Task 1 - CI/CD Pipeline + SAST

4.1 Yêu cầu kỹ thuật

Yêu cầuCách triển khai
Pipeline tối thiểu gồm Build, Test, SecurityĐược thực hiện bởi build_target.py, test_target.py và nhóm script scan_*.py.
Optional deploy mockKhông triển khai deploy mock; phạm vi repo tập trung vào scan, policy và reporting.
Tích hợp SAST scanDùng Semgrep qua scan_sast.py.
Sinh report scanXuất report dưới dạng SARIF/JSON trong security-results/.
Pipeline parse report để ra pass/failreport_generator.py hợp nhất finding; policy_engine.py dùng report này để ra quyết định.

4.2 Phân tích

Câu hỏiTrả lời
Vì sao chọn SAST tool này?Chọn Semgrep vì nhẹ, dễ chạy trong container, không cần server riêng và dễ xuất SARIF/JSON để parse.
SAST chạy ở stage nào? Vì sao?Chạy sau Build/Test và trước các bước scan khác để phát hiện lỗi source code sớm, giảm chi phí sửa lỗi.
Issue nào block, issue nào warning?Critical block ngay; từ 2 High trở lên block; Medium warning; Low allow và đưa vào backlog.

4.3 Hướng dẫn chạy demo

  2. Mở dashboard và chọn target juice-shop.
  3. Chạy scan và chờ pipeline hoàn tất.
  4. Mở security-results/sast_results.sarif hoặc report tương ứng để xem finding SAST.
  5. Kiểm tra quyết định pass/fail trong policy_result.json và trên dashboard.

6. Task 3 - Policy Enforcement & Security Governance

6.1 Scenario bắt buộc

Scenario yêu cầu pipeline phát hiện: 1 Critical, 2 High, 2 Medium.

Với policy hiện tại của AegisFlow, scenario này sẽ bị BLOCKED vì có Critical và cũng vượt ngưỡng High.

6.2 Thiết kế policy enforcement

IF critical_count > 0 THEN BLOCK ELSE IF high_count >= 2 THEN BLOCK ELSE IF risk_weight > 50 THEN WARNING ELSE PASS
Nội dungThiết kế
Điều kiện block / allowCritical block; từ 2 High trở lên block; Medium/Low theo warning hoặc allow.
Exception có cho phép không?Có, nhưng phải có owner, thời hạn và lý do.
Ai approve?AppSec Lead + Engineering Manager; có thể cần Product Owner tùy mức ảnh hưởng release.

6.3 Dashboard / reporting

Nhóm người xemThông tin cần nhìn thấy
CISO / BODSecurity score, số lượng Critical/High, trạng thái pass/fail, audit trail.
Dev teamFinding cụ thể, file/line, root cause, remediation.
ProductẢnh hưởng tới release, SLA và exception status.

6.4 Quy trình xử lý

Câu hỏiTrả lời
Ai chịu trách nhiệm fix?Dev owner là người sửa chính; AppSec hỗ trợ triage và xác nhận risk.
SLA cho Critical / High?Critical: 24 giờ. High: 72 giờ.
Giao tiếp với Product như thế nào?Thông báo tác động tới release, xác nhận deadline remediation hoặc risk acceptance.

  4. Mô tả trên dashboard nhóm thông tin dành cho CISO/BOD, Dev team và Product.

7. Task 4 - Security of the Pipeline

7.1 Yêu cầu

Phân tích bản thân pipeline như một hệ thống cần được bảo vệ và liên hệ trực tiếp với pipeline đã build ở Task 1-2.

7.2 Threat -> Impact -> Mitigation

ThreatImpactMitigation trong pipeline
Secrets leakageLộ API key hoặc token.Gitleaks, env var, không commit secret vào repo.
Compromised CI runnerSửa artifact hoặc đánh cắp biến môi trường.Runner container hóa, giảm privilege, audit log.
Malicious pipeline modificationBypass scanner hoặc quality gate.Code review cho pipeline/ và CI config.
Over-permissioned service accountToken có quyền quá rộng.Least privilege, token read-only mặc định.
Supply chain attackTool hoặc dependency build bị thay thế độc hại.Pin version scanner, SBOM, quét dependency.

7.3 Demo minh họa

  1. Dùng cùng pipeline ở Task 1-2 để chạy secret scan, SCA, IaC scan và policy.
  2. Kiểm tra artifact trong security-results/ để chỉ ra rủi ro đã được phát hiện ở đâu.
  3. Đối chiếu mitigation tương ứng trong Dockerfile, compose hoặc pipeline scripts.

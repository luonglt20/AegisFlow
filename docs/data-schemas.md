# AegisFlow Data Schemas & Contracts

To fully understand how AegisFlow components communicate without reading the source code, you must understand its core JSON data contracts. The entire system is decoupled via JSON files written to the `security-results/` directory.

## 1. The Unified Finding Schema (`full_report.json`)
This is the core data structure generated by `report_generator.py`. It normalizes outputs from all disparate tools (Semgrep, Trivy, Checkov, etc.) into a single array of findings.

```json
{
  "scan_metadata": {
    "app_name": "JuiceShop",
    "security_score": 65,
    "pipeline_run_id": "pipeline-run-20260430120000",
    "total_findings": 12,
    "by_severity": { "CRITICAL": 1, "HIGH": 3, "MEDIUM": 8, "LOW": 0 }
  },
  "findings": [
    {
      "id": "FIND-001",
      "source_tool": "Semgrep",
      "scan_type": "SAST",
      "rule_id": "javascript.express.security.audit.sqli",
      "title": "SQL Injection Detected",
      "severity": "CRITICAL",
      "cvss_v3": 9.8,
      "cve_cwe": "CWE-89",
      "owasp_2025": "A03:2025 – Injection",
      "mitre_attack": {
        "tactic": "Initial Access",
        "technique": "Exploit Public-Facing Application"
      },
      "affected_file": "/app/src/db.js",
      "affected_line": 42,
      "code_snippet": "db.query(`SELECT * FROM users WHERE id = ${req.body.id}`);",
      "status": "PENDING_HUMAN_VERIFY"
    }
  ]
}
```

## 2. The AI Triaged Output (`full_report_triaged.json`)
The `ai_triage_engine.py` reads `full_report.json` and enriches the findings. It modifies the existing findings array by adding AI-specific fields.

```json
{
  "...": "Inherits everything from full_report.json, plus:",
  "findings": [
    {
      "status": "AI_TRIAGED",
      "ai_analysis": "The code directly interpolates user input into the SQL query without parameterization, leading to an immediate SQLi vector.",
      "ai_fix": {
        "before": "db.query(`SELECT * FROM users WHERE id = ${req.body.id}`);",
        "after": "db.query('SELECT * FROM users WHERE id = ?', [req.body.id]);",
        "explanation": "Used parameterized queries to sanitize input."
      }
    }
  ]
}
```

## 3. The Policy Result (`policy_result.json`)
Generated by `policy_engine.py`, this dictates the final pipeline outcome.

```json
{
  "timestamp": "2026-04-30T12:05:00Z",
  "pipeline_status": "BLOCKED",
  "passed": false,
  "block_reason": "1 CRITICAL finding(s) detected — pipeline gate enforced",
  "findings_summary": {
    "total": 12,
    "critical": 1,
    "high": 3,
    "medium": 8,
    "low": 0
  },
  "quality_summary": {
    "build": { "status": "passed", "mode": "real" },
    "test": { "status": "skipped", "mode": "validation" }
  }
}
```

## 4. The Audit Log (`audit_log.json`)
Generated by `audit_logger.py`, this is an append-only array.

```json
[
  {
    "timestamp": "2026-04-30T12:05:01Z",
    "event_type": "PIPELINE_EXECUTION",
    "user": "local-operator",
    "pipeline_id": "AF-260430-12345",
    "outcome": "BLOCKED",
    "integrity_hash": "sha256:d8a2...[truncated]...e9f1"
  }
]
```

## Why This Matters for AI
By understanding these schemas, an AI assistant can:
- Write new parsers (e.g., adding a SonarQube parser) simply by mapping it to the `full_report.json` schema.
- Modify the Dashboard UI by knowing exactly what fields exist in the JSON payload.
- Adjust the AI Triage prompts to inject new fields into the `ai_fix` object.

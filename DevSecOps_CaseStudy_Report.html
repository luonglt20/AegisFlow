Cách Run Demo

  1. Bật Docker Desktop.
  2. Chạy docker compose up --build hoặc bash run_mac.sh.
  3. Mở http://localhost:58081.
  4. Chọn folder cần scan. Mặc định repo đang mount ./real-apps vào /app/workspace.
  5. Nhấn Scan, đợi pipeline hoàn thành, rồi xem kết quả ở dashboard và thư mục security-results/.

Kiến trúc và Toolchain

Toolchain trong artifact hiện tại là: Semgrep, Trivy, Gitleaks, Checkov, Nuclei, Syft. Với DAST, runtime scanner trong repo là Nuclei, còn report hợp nhất hiện đang được normalize theo model artifact OWASP ZAP style report.

Task 1 – CI/CD Pipeline + SAST

Pipeline được thể hiện ở hai lớp: lớp local orchestrator là pipeline/run_pipeline.sh và lớp CI artifact là .gitlab-ci.yml. Các stage tối thiểu đều có mặt: build, test, security, report.

Thiết kế Stage

Build Validation -> Test Validation -> SAST -> SCA -> SBOM -> Secrets -> IaC -> DAST -> Report -> AI Triage -> Policy Gate

Vì sao chọn Semgrep cho SAST

  • Cài đặt nhanh trong container Python, phù hợp bài toán local demo bằng Docker Compose.
  • Hỗ trợ nhiều ngôn ngữ trong cùng repo, quan trọng vì repo chứa Node.js, Python, PHP và các sample app khác nhau.
  • Xuất SARIF/JSON dễ parse để hợp nhất vào full report và policy gate.

Stage chạy SAST và lý do

SAST chạy ngay sau build/test validation. Lý do: lỗ hổng trong code nên được phát hiện sớm nhất có thể, trước khi đi tới dependency, infrastructure và runtime scanning. Với kiến trúc này, Semgrep là security gate đầu tiên trong pipeline.

Task 3 – Policy Enforcement & Security Governance

Scenario bắt buộc của đề bài được đáp ứng và còn vượt ngưỡng: run hiện tại có 0 Critical, 0 High, 0 Medium, 0 Low. Vì vậy pipeline kết thúc ở trạng thái UNKNOWN với lý do: See findings summary.

Logic policy

If CRITICAL > 0: BLOCK
Else if HIGH >= 2: BLOCK
Else if weighted risk > threshold: WARNING
Else: PASS

Dashboard / Reporting View

Đối tượng xemThông tin chính
CISO / BODPipeline status, severity distribution, số findings theo scan type, audit trail, maturity/compliance mapping.
Dev / App TeamFile và line bị ảnh hưởng, CVSS, CWE/CVE, MITRE ATT&CK, remediation hint, AI triage, evidence snippet.

Quy trình xử lý và SLA

  • Owner chính là team phát triển của ứng dụng được scan; security team giữ vai trò gatekeeper và reviewer.
  • Critical: xử lý trong 24 giờ ở policy layer, AI triage hiện tại còn dùng deadline ngắn hơn cho dashboard.
  • High: xử lý trong 72 giờ theo policy.
  • Product/Engineering manager cần được thông báo khi pipeline bị block để quyết định release hold, exception hoặc hotfix window.

Exception và approval

Project có mô hình exception ở mức dữ liệu/reporting: finding giữ cờ exception_approved, audit log lưu actor/action, nhưng chưa có workflow UI hoàn chỉnh để enforce multi-approver production-grade. Đây là điểm phù hợp để giải thích trong case study như một giới hạn hiện tại.

Task 5 – Secure-by-Design & Threat Modeling

Task 5 được mô tả như một hệ thống khác với target app scan thực tế, đúng yêu cầu đề bài. Mục tiêu của phần này là chứng minh tư duy secure-by-design chứ không phải thêm một pipeline thứ hai.

Web App -> API -> Database -> External Payment Service
ThreatDesign ControlCI/CD Security Check Mapping
Broken authentication / API abuseJWT validation, RBAC, MFA cho action nhạy cảmSAST rules cho auth bypass, API review gate và secret scan cho token handling
SQL injection vào payment recordsParameterized queries, centralized validation, allowlist schemaSAST block rule + runtime DAST validation
Secret leakage sang image hoặc repoSecret manager, rotation policy, zero hardcoded credentialsGitleaks + Trivy/SCA + SBOM review

Task 1 - CI/CD Pipeline + SASTTrang 2/5

Task 1. CI/CD Pipeline + SAST

Pipeline tối thiểu gồm Build, Test, Security và Report. Cấu hình CI tham chiếu nằm ở .gitlab-ci.yml; local runner nằm ở pipeline/run_pipeline.sh. Cách này cho phép người chấm chạy bằng Docker nhưng vẫn thấy được tư duy CI/CD thật.

Flow thực thi chi tiết

1. build_target.py : phát hiện stack, validate build script hoặc chạy build thật khi có cấu hình 2. test_target.py : chạy test thật nếu có lệnh portable; nếu không, ghi rõ validation mode 3. scan_sast.py : Semgrep scan, xuất SARIF/JSON 4. scan_sca.py : Trivy filesystem dependency scan 5. scan_sbom.py : sinh SBOM để truy vết supply chain 6. scan_secret.py : Gitleaks phát hiện secret hardcoded 7. scan_iac.py : Checkov kiểm tra Docker/IaC 8. scan_dast.py : Nuclei/DAST live hoặc predictive mode 9. report_generator.py : hợp nhất report 10. ai_triage_engine.py : triage ngữ cảnh, fallback deterministic khi không có GROQ_API_KEY 11. policy_engine.py : quyết định pass/warn/block

Lý do chọn SAST tool

Công cụLý do chọnVị trí trong pipeline
SemgrepNhẹ, chạy tốt trong container, hỗ trợ nhiều ngôn ngữ, dễ xuất SARIF/JSON và phù hợp repo nhiều sample app.Sau Build/Test, trước SCA/IaC/DAST để phát hiện lỗi code sớm.

Giả lập và giới hạn

Build/Test có thể chạy ở validation mode khi target không có dependency hoặc command portable. Đây không phải build production đầy đủ; mục tiêu là chứng minh orchestration, artifact và policy flow. Khi đưa vào CI thật, bật cài dependency theo stack và chạy test suite đầy đủ.

Container / IaC logic

Pipeline không chỉ tìm CVE mà còn kiểm tra "đường đi tới production": Dockerfile có healthcheck hay không, workflow có permission quá rộng hay không, secret có bị lộ trước khi build image hay không. Với container image scanning production, bước tiếp theo là thêm trivy image sau khi build image thay vì chỉ trivy fs.

Task 3. Policy Enforcement & Security Governance

Scenario đề bài yêu cầu pipeline phát hiện 1 Critical, 2 High, 2 Medium. Policy của repo còn chặt hơn: chỉ cần có Critical là block; từ 2 High trở lên cũng block. Report mẫu hiện tại có 15 Critical, 39 High, 48 Medium nên trạng thái là BLOCKED.

Sơ đồ logic policy

IF critical_count > 0 THEN BLOCK, owner must fix or request exception ELSE IF high_count >= 2 THEN BLOCK ELSE IF risk_weight > 50 THEN WARNING, release allowed only with ticket ELSE PASS
Điều kiệnDecisionExceptionApprover
Critical bất kỳBlockChỉ cho phép khi false positive đã chứng minh hoặc compensating control tạm thời.AppSec Lead + Engineering Manager, CISO nếu production impact.
Từ 2 High trở lênBlockCó expiry date, ticket, owner, rollback plan.AppSec Lead + Product Owner.
Medium/LowWarning/AllowKhông cần exception formal nếu đã có backlog và SLA.Team Lead.

Task 5 - Secure-by-Design & Threat ModelingTrang 5/5

Task 5. Secure-by-Design & Threat Modeling

Task 5 dùng một hệ thống khác với target scan của Task 1-4: Web App + API + Database + External Payment Service. Mục tiêu là chứng minh tư duy thiết kế bảo mật trước khi code.

DFD mức cao

[Customer Browser] | HTTPS + session/JWT v [Web App / Frontend] | REST/JSON + access token v [Backend API] ---- SQL/TLS ----> [Database] | | signed payment request + idempotency key v [External Payment Service] | v [Payment Webhook Endpoint] -> verify signature -> update order status

Threat mapping

ThreatDesign controlCI/CD security check
API/auth: attacker dùng JWT giả, token hết hạn nhưng vẫn accepted hoặc thiếu authorization theo object.OIDC/JWT verification, short-lived token, RBAC/ABAC, object-level authorization, deny-by-default middleware.SAST rule cho missing auth middleware, unit/integration test authz, DAST test endpoint unauthenticated.
Payment tampering: sửa amount/order_id hoặc replay webhook.Server-side price calculation, signed request, webhook signature verification, idempotency key, replay window.SAST tìm thiếu signature verification, API test replay/idempotency, secret scan cho payment key.
Data disclosure: SQL injection hoặc database exposed/misconfigured.Parameterized query/ORM safe mode, least privilege DB user, encryption at rest/in transit, private network.SAST injection rules, SCA database driver CVE scan, IaC scan public exposure/security group.

Liên kết threat model với pipeline

Threat model không dừng ở sơ đồ. Mỗi control phải có một check trong CI/CD: auth control được kiểm bằng SAST và test; payment control được kiểm bằng test chữ ký/webhook; database control được kiểm bằng SAST, SCA và IaC scan. Nhờ vậy thiết kế bảo mật được chuyển thành policy tự động, tránh tình trạng chỉ có tài liệu nhưng không có enforcement.

Kết luận

AegisFlow đáp ứng yêu cầu thực hành bằng một hệ thống có thể chạy lại local, có artifact rõ ràng, có governance và có phân tích bảo mật của chính pipeline. Các phần giả lập/fallback được giới hạn ở build/test portability, AI triage khi thiếu API key và DAST predictive mode khi không có URL live; các giới hạn này đã được nêu rõ để người chấm phân biệt demo orchestration với production CI hoàn chỉnh.
